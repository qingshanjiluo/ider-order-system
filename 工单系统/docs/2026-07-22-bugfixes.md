### 修复 5：营收将灵石/修仙币当做人民币计算

**根因**：[`admin/stats.js`](../functions/api/admin/stats.js) 使用 `SUM(bonus_points)` 或仅统计 `payment_method = 'wechat'` 的订单来计算营收，导致：
- 灵石支付的订单被忽略（如果只统计 wechat）
- 修仙币支付的订单的 `bonus_points` 被直接相加（不同单位混算）

**修复**：引入 `REVENUE_TO_RMB` CASE 表达式，按支付方式将 price 统一转换为人民币：

| 支付方式 | price 存储单位 | 转换公式 |
|---------|-------------|---------|
| `wechat` | 人民币（元） | 直接使用 `price` |
| `coin` | 修仙币数量 | `price / 120.0` |
| `spirit_stone` | 万灵石 | `price × 120 × 10000 / spirit_stone_per_10_points` |

三个统计查询均已修复：
- 总营收 `totalRevenue`
- 今日营收 `todayRevenue`
- 近7日趋势 `dailyTrend`

### 修复 6：用户修仙币异常增加

**根因**：[`orders/[id]/status.js`](../functions/api/orders/[id]/status.js) 审核通过时，`total_spent` 使用 `order.bonus_points` 更新：

```javascript
// 修复前 — bonus_points 单位因支付方式不同而不同
UPDATE users SET total_spent = total_spent + bonus_points
```

对于不同支付方式：
- `wechat`：bonus_points 是积分数量（如120），而非人民币（如1元）
- `coin`：bonus_points 是修仙币数量
- `spirit_stone`：bonus_points 是积分数量

这导致 `total_spent` 字段数值远大于实际消费。

**修复**：按支付方式将 price 转换为人民币等价值后更新 `total_spent`：

```javascript
// 修复后 — 统一使用人民币单位
let spentRMB = 0;
if (order.payment_method === 'wechat') {
  spentRMB = order.price;  // 已是人民币
} else if (order.payment_method === 'coin') {
  spentRMB = order.price / 120;  // 修仙币 → 人民币
} else if (order.payment_method === 'spirit_stone') {
  // 万灵石 → 人民币（需查 config）
  spentRMB = order.price * 120 * 10000 / spiritPer10;
}
UPDATE users SET total_spent = total_spent + spentRMB
```

### 修复 7：兑换码同一用户可多次使用

**根因**：[`redeem/index.js`](../functions/api/redeem/index.js) 中 `redeem_log` 表用于追踪每个用户的兑换记录，但：

1. [`schema.sql`](../schema.sql) 中 `redeem_log` 表只有 `xp` 列，缺少 `coins` 列
2. 代码中 INSERT 使用 `coins` 列：`INSERT INTO redeem_log (user_id, code, coins)`
3. 如果表由 schema.sql 创建（无 coins 列），INSERT 会静默失败
4. 后续的重复使用检查 `SELECT id FROM redeem_log WHERE user_id = ? AND code = ?` 查不到记录，导致同一用户可重复兑换

**修复**：
1. [`schema.sql`](../schema.sql) 的 `redeem_log` 表添加 `coins INTEGER DEFAULT 0`
2. [`redeem/index.js`](../functions/api/redeem/index.js) 添加动态迁移：
   ```javascript
   // 兼容旧表：如果 redeem_log 已存在但缺少 coins 列，动态添加
   try {
     await env.DB.prepare("ALTER TABLE redeem_log ADD COLUMN coins INTEGER DEFAULT 0").run();
   } catch (e) { /* 列已存在 */ }
   ```

### 修复 8：工单提交错误 500

**根因**：[`orders/index.js`](../functions/api/orders/index.js) 的 INSERT 语句使用了 6 个 `schema.sql` 中不存在的列：

| 列名 | 用途 | schema 中是否存在 |
|------|------|:---:|
| `frozen_points` | 冻结的修仙币 | ❌ |
| `invite_code_used` | 使用的邀请码 | ❌ |
| `game_account_name` | 游戏账号名 | ❌ |
| `game_account_password` | 游戏账号密码 | ❌ |
| `subscription_start` | 订阅开始时间 | ❌ |
| `subscription_end` | 订阅结束时间 | ❌ |

D1 数据库执行 INSERT 时因列不存在而返回 500 错误。

**修复**：
1. [`schema.sql`](../schema.sql) 的 `orders` 表添加全部 6 个缺失列
2. [`orders/index.js`](../functions/api/orders/index.js) POST 处理器添加动态迁移：
   ```javascript
   const ADD_COLUMNS = [
     'frozen_points INTEGER DEFAULT 0',
     'invite_code_used TEXT DEFAULT ""',
     'game_account_name TEXT DEFAULT ""',
     'game_account_password TEXT DEFAULT ""',
     'subscription_start TEXT DEFAULT ""',
     'subscription_end TEXT DEFAULT ""',
   ];
   for (const col of ADD_COLUMNS) {
     try { await env.DB.prepare(`ALTER TABLE orders ADD COLUMN ${col}`).run(); } catch (e) { /* 列已存在 */ }
   }
   ```
3. 创建迁移 SQL 文件 [`fix_missing_columns.sql`](./fix_missing_columns.sql) 供手动执行

## 部署注意

1. **自动迁移**：修复 7 和 8 在代码中包含动态 `ALTER TABLE`，部署后首次请求会自动添加缺失列，无需手动执行 SQL
2. **手动迁移（可选）**：如需提前执行，可在 D1 控制台运行 [`fix_missing_columns.sql`](./fix_missing_columns.sql)
3. **历史数据**：`total_spent` 字段中已存在的错误数据（基于 bonus_points 而非人民币）不会自动修正。如需修正历史数据，需运行数据修复脚本

### 修复 9：近7日收入趋势包含被拒绝工单

**根因**：[`admin/stats.js`](../functions/api/admin/stats.js) 的 `dailyTrend` 查询缺少 `status` 过滤条件：

```sql
-- 修复前 — 所有状态的工单都计入收入趋势
WHERE created_at >= datetime('now', '-7 days')

-- 修复后 — 仅统计 approved/completed/active
WHERE created_at >= datetime('now', '-7 days') AND status IN ('approved','completed','active')
```

总营收和今日营收查询已有正确的状态过滤，但近7日趋势查询遗漏了。

### 修复 10：删除用户接口 500 错误

**根因**：[`admin/users/[id]/delete.js`](../functions/api/admin/users/%5Bid%5D/delete.js) 直接删除用户记录，但数据库中多个表通过 `FOREIGN KEY (user_id) REFERENCES users(id)` 引用了 users 表：

| 引用表 | 外键列 |
|--------|--------|
| sessions | user_id |
| orders | user_id |
| notifications | user_id |
| appeals | user_id |
| bot_logs | user_id |
| checkin_logs | user_id |
| order_activities | user_id |
| redeem_log | user_id |
| withdrawals | user_id |

SQLite 外键约束导致删除用户时，如果关联表中存在记录，DELETE 操作会失败并返回 500 错误。

**修复**：在删除用户前，先清理所有关联表中的数据：

```javascript
// 修复后 — 先删除关联数据，再删除用户
for (const table of USER_RELATED_TABLES) {
  await env.DB.prepare(`DELETE FROM ${table} WHERE user_id = ?`).bind(targetId).run();
}
// 处理 orders 关联的 game_accounts
const orders = await env.DB.prepare('SELECT id FROM orders WHERE user_id = ?').bind(targetId).all();
for (const order of orders.results || []) {
  await env.DB.prepare('DELETE FROM game_accounts WHERE order_id = ?').bind(order.id).run();
}
await env.DB.prepare('DELETE FROM orders WHERE user_id = ?').bind(targetId).run();
await env.DB.prepare('DELETE FROM coupons WHERE created_by = ?').bind(targetId).run();
await env.DB.prepare('DELETE FROM users WHERE id = ?').bind(targetId).run();
```
